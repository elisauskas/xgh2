About XGH 2nd Edition

The XGH (eXtreme Go Horse) methodology was originally conceptualized by Brazilian developers and documented in the legendary 22 Axioms of the Extreme Go Horse Methodology. It revolutionized software development by acknowledging what every developer knew but few dared to say: sometimes you just need to get things done, regardless of best practices.

This second edition builds upon those foundational principles while incorporating the realities of modern software development. We now live in an era of cloud-native applications, microservices architectures, AI-assisted coding, and most importantly, vibe-driven development.

The original XGH philosophy emerged from the trenches of real-world software development, where developers learned that pragmatism often trumps perfection. This updated edition expands from 22 to 23 axioms, adding modern context for the post-AI era while maintaining the irreverent spirit that made XGH a cult classic among developers worldwide.

Whether you're dealing with Kubernetes clusters that have achieved sentience, microservices that communicate through passive-aggressive error messages, or stakeholders who change requirements faster than you can implement them, these axioms will guide you through the chaos with humor and hard-won wisdom.

The 23 Axioms

Axiom 1
I think, therefore it's not XGH. In XGH you don't think, you do the first thing that comes to your mind. There's not a second option as the first one is faster.
Axiom 2
There are 3 ways of solving a problem: the right way, the wrong way, and the XGH way; which is exactly like the wrong one but faster.
Axiom 3
Vibe-driven development trumps test-driven development. If the code feels right, it probably is.
Axiom 4
XGH is completely reactive. Errors only come to exist when someone notices them.
Axiom 5
In XGH anything goes. It solves the problem? It compiled? You commit it and don't think about it anymore.
Axiom 6
You commit always before updating. If things go wrong your part will always be correct... and your colleagues will be the ones dealing with the problems.
Axiom 7
Copy-paste from AI chat is not just allowed, it's encouraged. Why reinvent the wheel when an AI already invented a perfectly good wheel for you?
Axiom 8
XGH doesn't have schedules. You will ALWAYS be able to implement EVERYTHING in time (even if that means accessing the database through some crazy script).
Axiom 9
You'll always need to do more and more XGH. For every bug solved using XGH, seven more are created. As all of them will be solved using XGH, its utility tends to the infinite.
Axiom 10
XGH is absolute. Delivery dates and costs are absolute things. Quality is relative. Never think about quality but instead think about the minimum time required to implement a solution.
Axiom 11
Tests are for people who don't trust their AI copilot. If you know what you're doing, then why test? If it compiles, it's good.
Axiom 12
There's no refactoring, just rework. If things ever go wrong just use XGH to quickly solve the problem. When the problem requires rewriting the whole software it's time to drop off.
Axiom 13
Be used to the 'living on the edge' feeling. The project failed? You learned something with it? Then for you it was a success!
Axiom 14
If it's working don't bother. Never ever change‚Äîor even think of questioning‚Äîa working code. That's a complete waste of time.
Axiom 15
A good developer codes with their gut and Claude. A great developer codes with their gut, Claude, and a junior developer to blame.
Axiom 16
Be ready to jump ship or blame someone else when it goes down. For people using XGH someday the ship goes down. When that day comes, you'd better have your resume ready.
Axiom 17
Don't try to go against the tide. If your colleagues use XGH and you are the only one who wants to do things the right way then quit it!
Axiom 18
The problem is only yours when your name is on the code docs. Never touch code which you're not the author. When a team member leaves, it will eventually go down.
Axiom 19
XGH is anarchic. There's no need for a project manager. There's no owner and everyone does whatever they want when problems and requirements appear.
Axiom 20
Be authentic. XGH doesn't follow patterns. Write code as you see fit. If it solves the problem, commit and forget about it.
Axiom 21
Always believe in improvement promises. Putting TODO comments in the code helps the XGH developer not feel guilt. Surely there won't be any refactoring.
Axiom 22
XGH is your bro. But it's vengeful. If you start with XGH and then turn to some trendy methodology you will be screwed. XGH doesn't allow refactoring.
Axiom 23
XGH is not dangerous until you see some order in it. A XGH project is always in chaos. Don't try to put order into XGH as it's useless and will make things go down faster.
